==================================================================
CORE INVARIANT
==================================================================
The daemon never runs a model. Fable runs every model. The app is a
thin client. The daemon owns queue state, SQLite persistence, git
worktree lifecycle, advisory write-locks, the session registry, SSE
fan-out, and JSON-Schema validation — it touches git + fs but never an
LLM. Fable (arc-orchestrator in a live Claude Code session) does all
model work and PULLS work; the daemon never pushes. The app renders and
turns gestures into MCP calls.

Design record: ../BUILD_SPEC.md. Prototype (layout SoT): ../prototype/Story Queue.dc.html.
Design system: ../DESIGN_SYSTEM.md + app/src/tokens.css (used verbatim).

==================================================================
GITHUB FILING FLOW (design — Fable half is model/CLI work, not daemon code)
==================================================================
Buildable seam (DONE): a draft carries fileRequested; story.requestFile flags it and
emits a "file-requested" event; file.pending(projectId?) is the pull queue; story.file(id,issue)
clears draft+flag, sets issue, emits "filed". Board drawer offers "Request Fable to file"
and a manual "File now" (deterministic fallback for local / no-Fable).

Fable half — BUILT as a standalone bridge (mcp-server/file-agent.ts, `npm run file-agent`):
  connects as an MCP client → file.pending → gh issue create (deterministic body via
  buildIssueBody, no model) → story.file(id, issue-url) → guardrail lifts. Issue-creator is
  injectable (tested with a fake; runs real `gh` by default). A live Fable session can run
  this same tool, or it runs standalone. `--dry-run` reports without creating issues.
  Deterministic fallback stays the drawer's "File now" for offline/local use.
  NOTE: a non-dry run creates REAL GitHub issues — run intentionally against the right repo.
